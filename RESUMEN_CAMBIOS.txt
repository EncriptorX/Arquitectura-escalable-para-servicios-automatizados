================================================================================
ARQUITECTURA FINAL
================================================================================

FLUJO COMPLETO:

Cliente → cliente-abc123.suncarsrl.com → Backend Proxy → www.cliente.com
          (Custom Hostname)              (api/proxy.py)   (Origin Real)

COMPONENTES:

1. Custom Hostname (Cloudflare)
   - Subdominio único bajo suncarsrl.com
   - SSL/TLS automático (DV por HTTP)
   - SIN custom_origin_server ni custom_origin_sni

2. Backend Proxy (Python)
   - Lee header Host de cada request
   - Identifica subdominio
   - Busca dominio real en mapa en memoria
   - Reenvía solicitud al origin
   - Devuelve respuesta al cliente

3. Mapa en Memoria
   - ProxyConfig.SUBDOMAIN_MAP
   - {subdominio: dominio_real}
   - Sincronizado con CSaaSConfig.PROVISIONED_CLIENTS

INSTRUCCIONES PARA EL CLIENTE
================================================================================

OPCIÓN 1: Usar subdominio directamente
- URL: https://cliente-abc123.suncarsrl.com
- Sin cambios DNS
- Funciona inmediatamente

OPCIÓN 2: Mantener dominio original (RECOMENDADO)
- Crear registro CNAME en DNS:
  Tipo: CNAME
  Nombre: www
  Valor: cliente-abc123.suncarsrl.com
  TTL: 3600
- Esperar propagación DNS (5-30 minutos)
- Resultado: www.cliente.com muestra contenido protegido

================================================================================
LIMITACIONES CONOCIDAS
================================================================================

🔴 ALMACENAMIENTO EN MEMORIA
   Problema: Los mapeos se pierden al reiniciar el servidor
   Solución Futura: Implementar base de datos (Supabase, PostgreSQL)

🟡 LATENCIA ADICIONAL
   Problema: El proxy agrega un salto adicional (~50-200ms)
   Mitigación: Usar CDN de Cloudflare para cachear contenido estático

🟡 CONFIGURACIÓN DE VERCEL
   Problema: El proxy debe manejar todas las solicitudes a subdominios
   Solución: Configurar Vercel correctamente o usar dominio dedicado
